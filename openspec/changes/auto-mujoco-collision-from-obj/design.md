## Context

当前 MuJoCo 地形加载中，`_create_trimesh()` 方法（`scene_manager.py:209-268`）对 `load_obj` 类型地形的处理方式是：
- OBJ mesh 仅作为视觉几何（`contype=0, conaffinity=0`）
- 碰撞由一个硬编码的 ground plane + 一个硬编码的 box geom 代替
- 硬编码 box 的参数（pos、size）需要手动计算并维护

`terrain_climb_14_50cm.obj` 包含两个障碍物：
1. 不规则四边形底面的 0.5m 高 climb box
2. 5x5m 的 0.1m 高条形平台

当前 MuJoCo 只为障碍物 1 提供了碰撞（矩形近似），障碍物 2 完全缺失。

## Goals / Non-Goals

**Goals:**
- 从 OBJ 文件自动提取障碍物几何并生成 MuJoCo box 碰撞体
- 支持多个障碍物的自动检测和分解
- 移除硬编码碰撞参数，实现地形文件变更时的零维护

**Non-Goals:**
- 不追求像素级精度的碰撞近似（box 近似足够用于策略迁移）
- 不修改 IsaacSim 侧的地形加载逻辑
- 不支持非凸多边形的精确碰撞（MuJoCo 限制）

## Decisions

### 决策 1：障碍物提取策略

**方案 A（推荐）：基于 Z 高度分层提取**
- 将 OBJ mesh 按 Z 坐标分层，识别高于地面（Z > 阈值）的独立区域
- 每个独立区域计算 bounding box，生成 MuJoCo box geom
- 优点：简单可靠，自动处理任意数量障碍物
- 缺点：相邻障碍物可能被合并

**方案 B：基于连通组件分析**
- 构建 mesh 的面邻接图，按 Z > 阈值的面进行连通组件分析
- 每个连通组件独立计算 bounding box
- 优点：精确分离相邻障碍物
- 缺点：实现复杂，依赖图算法

**选择方案 A**：对于当前地形文件，障碍物在空间上是分离的，Z 高度分层足够。如果未来需要更精确的分离，可以升级到方案 B。

### 决策 2：碰撞近似方式

**方案 A（推荐）：Bounding box 近似**
- 每个障碍物用一个轴对齐的 box 近似
- 实现简单，性能好
- 对于 climb 场景，机器人主要接触顶面，侧面精度要求不高

**方案 B：多 box 拼接**
- 将不规则四边形分解为 2-3 个矩形区域
- 更精确但实现复杂
- 需要凸分解算法

**选择方案 A**：bounding box 近似对于策略迁移已足够。如果 sim2sim 验证发现精度不足，可作为后续优化。

### 决策 3：阈值参数化

- `ground_z_threshold`：区分地面和障碍物的 Z 高度阈值（默认 0.01m）
- `min_obstacle_height`：最小障碍物高度（忽略低于此值的特征）
- 这些参数应可通过 terrain config 传入

## Risks / Trade-offs

- **[风险] 近似精度不足** → 通过 sim2sim 验证对比，如不足则升级为多 box 拼接
- **[风险] 障碍物合并** → 相邻障碍物可能被识别为一个，需要调整阈值或升级为连通组件分析
- **[风险] 非轴对齐障碍物** → bounding box 对斜向障碍物会多覆盖面积，当前 climb 场景可接受
- **[权衡] 通用性 vs 简单性** → 选择简单实现，优先覆盖当前用例，后续按需扩展
